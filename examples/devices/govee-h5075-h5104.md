# Govee H5075 & H5104 thermo-hygrometers (passive BLE advertisement)

- **Vendor app:** `com.govee.home` (Android; "Govee Home" on iOS)
- **HA integration:** native HA Core [`govee_ble`](https://www.home-assistant.io/integrations/govee_ble/)
  (+ [`govee-ble`](https://github.com/Bluetooth-Devices/govee-ble) parser). A worked clone built from
  this RE lives in [`examples/integrations/govee-passive-th/`](../integrations/govee-passive-th/).
- **Contributed by:** Claude (skill dry-run) · 2026-06-17

> **This is a different pattern from the rest of the library: a *passive advertisement* sensor.** The
> device **broadcasts** temperature/humidity/battery in its BLE advertisement — HA never connects
> (`connectable: false`). No command frame, no GATT write, no notify subscription. The whole
> integration is: match the advertisement → decode the manufacturer data → publish sensors.

## Transport

**Passive BLE advertisement (broadcast).** Confirmed three ways:
- HA Core `govee_ble` declares its `bluetooth` matchers `connectable: false` and never opens a GATT
  connection;
- the Govee app decompile parses these models from the advertisement (`*ThBroadParseImp` classes),
  not a connection;
- live capture: the values update purely from advertisements (`bluetooth/subscribe_advertisements`
  via any HA Bluetooth proxy hears them; no pairing/un-pairing needed).

**HA host:** any Bluetooth proxy (ESPHome/Shelly) or local adapter — passive scanning only.

## Connection / discovery keys

No connection. Discovery + decode key on:
- **local_name** prefix: `GVH5075_…`, `GVH5104_…`
- a 16-bit **service UUID `0xEC88`** present in the advertisement
- **manufacturer company id**: H5075 → `0xEC88`; H5104 → `0x0001`

## Status frame = the advertisement (golden frames, live-validated)

Both models pack temperature + humidity into one 3-byte big-endian word; battery is a separate byte:
```
value = u24_be(packed3)
temp_c = (value & 0x7FFFFF) // 1000 / 10      ; top bit 0x800000 = negative
hum_%  = (value & 0x7FFFFF) %  1000 / 10
batt_% = battery_byte & 0x7F                  ; bit 0x80 = error/low flag (govee-ble)
packed3 == FF FF FF  =>  no reading / fault
```
Where the bytes sit (within each model's manufacturer-data record, after the 2-byte company id):

| Model | company id | packed value | battery |
|---|---|---|---|
| **H5075** | `0xEC88` | `data[1:4]` | `data[4]` |
| **H5104** | `0x0001` | `data[2:5]` | `data[5]` |

**Golden frames (real captures, cross-checked):**
```
H5075  GVH5075_DC1B  mfr 0xEC88 = 0003ed733e00
       packed 03ed73=257395 -> 25.7°C (78.3°F) / 39.5% / batt 62%     [matched the device LCD]
H5104  GVH5104_6D7B  mfr 0x0001 = 010104a4652b
       packed 04a465=304229 -> 30.4°C (86.72°F) / 22.9% / batt 43%    [matched HA Core exactly]
```

## Home Assistant transition

The byte map becomes a pure `parse_advertisement()` feeding a `PassiveBluetoothProcessorCoordinator`
→ temperature/humidity/battery sensors. Full, unit-tested integration (golden-frame tests pass) in
[`examples/integrations/govee-passive-th/`](../integrations/govee-passive-th/):
```python
def parse_advertisement(manufacturer_data, *, local_name=None):
    data = manufacturer_data.get(0xEC88)          # H5075
    if data and len(data) >= 5:
        th = _decode_packed_th(data[1:4])         # (temp_c, hum)
        if th: return GoveeReading("H5075", *th, data[4] & 0x7F)
    data = manufacturer_data.get(0x0001)          # H5104
    if data and len(data) >= 6:
        th = _decode_packed_th(data[2:5])
        if th: return GoveeReading("H5104", *th, data[5] & 0x7F)
    return None
```

## Validation against the "answer key"

The app-derived decode was graded against the **merged HA Core `govee_ble` (`govee-ble==1.2.0`)** and
the **device LCD**, on live hardware: **identical** temperature/humidity for both models (battery
±1 %). The reverse-engineered byte map == shipping core == the physical display. (Read of `BleUtil`
confirmed the app's H5104 anchor — service-UUID `0xEC88` + company `0x0001` — maps onto `govee-ble`'s
offsets exactly; a previously-reported app-vs-library discrepancy, issue #110528, did **not**
reproduce.)

## Gaps & gotchas

- **Battery nuance:** the Govee app uses the full battery byte; `govee-ble` masks `& 0x7F` and treats
  bit `0x80` as an error/low flag. Use the masked form (we did). Accounts for the ±1 % seen.
- **Proxy-visible ≠ phone-visible:** during RE, an HA proxy heard a sensor at -47 dBm while the RE
  phone (a weak-radio handheld) heard nothing — capture from a good proxy/adapter, not a cheap phone.
- **H5127 is NOT in scope here.** Despite sharing the brand, **H5127 is a presence/motion sensor**
  (company `0x8803`-family) that the Govee app drives over a **GATT connection** (cmd type `0x21`,
  `ptReal` passthrough), while HA Core reads it from the advertisement — a *channel mismatch*. A live
  `0x8803`-family sibling showed advertisement byte 2 = `02`, while HA's matcher uses `EC 00 00 01`
  and the parser guards `EC 00 01 01` — a suspected HA-Core matcher/parser mismatch worth a separate
  investigation (needs a confirmed-owned H5127 to file).
